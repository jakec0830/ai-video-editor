# Windows 安裝流程紀錄

**日期**：2026-07-20
**回報人**：使用者（透過 Claude Code 全程 AI 代裝）
**環境**：Windows 11 Home 10.0.26200，全新環境（安裝前完全沒有 Node.js / Python / ffmpeg / git 以外的開發工具）
**安裝方式**：README「用桌面版 App / Claude Code 代裝」路徑，全程由 AI 執行 winget / pip / 手動腳本
**總耗時**：約 40 分鐘（含多次來回排除障礙，非單純下載等待時間）

這份紀錄的目的不是「東西壞了」，是把第一次在 Windows 上跑這個安裝流程時，AI 助手實際會撞到的每一個坑記下來，給開發者參考怎麼優化 `setup.sh` / README 的 Windows 安裝路徑，以及給「小白使用者」的引導話術。

## 一、環境檢查階段

### 問題 1：`python3` / `python` 在全新 Windows 上其實是空殼，不會如預期般失敗

Windows 內建的 App Execution Alias 讓 `python3`、`python` 指令在**沒有裝 Python** 的情況下依然「存在」（`command -v python3` 抓得到），只是執行下去會嘗試跳出 Microsoft Store 頁面，或直接無聲失敗。

`setup.sh` 第 14-18 行的檢查邏輯：
```bash
PY3="$(command -v python3 || true)"
if [ -z "$PY3" ]; then
  echo "$ERR 找不到 python3。..."
  exit 1
fi
```
在這台機器上**沒有正確擋下來**——`command -v python3` 找到的是 WindowsApps 底下的空殼執行檔，`$PY3` 不是空字串，所以腳本判斷「有 python3」直接往下跑，接著：
```bash
"$PY3" -m venv "$FREECUT/.venv"
```
這行**靜默失敗**（空殼不會真的建立 venv），腳本完全沒有報錯，一路跑到最後才在 `pip install` 那步噴出「No such file or directory」，而且錯誤指向 `.venv/Scripts/python.exe`，跟真正的病因（Python 根本沒裝）完全對不上，很容易誤導成是路徑設定問題。

**建議**：
- 偵測邏輯改成實際執行 `"$PY3" --version`，並檢查 stdout 是否真的吐出版本號（空殼通常吐空字串或直接無輸出/非 0 結束碼），而不是只看 `command -v` 找不找得到。
- venv 建立完立刻檢查 `Scripts/python.exe`（或 `bin/python3`）是否存在，不存在就在當下報錯退出，不要等到下一步 pip install 才爆炸。

### 問題 2：Windows 完全沒有自動安裝路徑，Mac 有 brew 一條龍，Windows 只有「檢查+提示」

`setup.sh` 對 Node / ffmpeg / 字型只做**檢查跟印提示文字**（`choco install ffmpeg` 之類），不像 Mac 那樣可以直接 `brew install` 全自動裝好。README 也是把 Windows 的「自動安裝」導向「叫 AI 代裝」這條路，但 AI 實際上沒有任何腳本可以照抄，得自己臨場判斷要用 winget、套件 ID 是什麼、裝完 PATH 什麼時候生效。

**這次實際用的 winget 套件 ID**（可行，供參考）：
| 依賴 | winget ID |
|---|---|
| Node.js LTS | `OpenJS.NodeJS.LTS` |
| ffmpeg | `Gyan.FFmpeg` |
| Python 3.12 | `Python.Python.3.12` |
| VC++ Redistributable | `Microsoft.VCRedist.2015+.x64` |

**建議**：新增一支 `setup.ps1`，用 winget 把 Node / ffmpeg / Python / VC++ Redist 都自動裝好（跟 `setup.sh` 對 Mac 用 brew 一樣的地位），而不是只讓 README 文字說「Windows 装 choco install ffmpeg」卻沒有真的腳本化。

## 二、字型安裝階段

### 問題 3：思源宋體在 Windows 上沒有套件管理員可以裝，只能手動下載+手動註冊登錄檔

Mac 是 `brew install --cask font-source-han-serif-vf` 一行搞定；Windows 完全沒有對應的套件（winget 搜尋不到任何 Source Han Serif 相關套件）。實際流程是：

1. 去 GitHub Releases API 找繁體中文子集的下載連結（`10_SourceHanSerifTC.zip`，約 132MB）
2. 下載、解壓縮
3. 用 PowerShell 逐一讀取每個 `.otf` 的 family name（`System.Drawing.Text.PrivateFontCollection`）
4. 手動寫入 `HKCU:\Software\Microsoft\Windows NT\CurrentVersion\Fonts` 登錄檔
5. 呼叫 `AddFontResource` + 廣播 `WM_FONTCHANGE` 訊息讓系統即時生效

這整段 README 只有一句話帶過（「其他系統到 github.com/adobe-fonts/source-han-serif 下載 .otf.ttc 安裝」），實際上是這次安裝過程中**最繁瑣、最容易出錯**的一步。

**過程中真的踩到的坑**：用 Windows Shell COM 物件（`Shell.Application` 的 `Namespace(0x14).CopyHere()`）裝字型時，因為同時手動複製了一次檔案到目的資料夾，導致同一個字型被裝了兩次、檔名多了 `_0` 後綴，登錄檔指向的檔案跟實際檔案對不起來，還一度誤刪了登錄檔正在參照的檔案，得整組砍掉重裝。另外多個字重（Bold/Regular）用 `.NET FontFamily.Name` 讀出來的名稱是同一個（都是 "Source Han Serif TC"），如果直接拿這個名稱當登錄檔 key，會互相覆蓋掉，只剩最後裝的那個字重是有效的。

**建議**：
- 開發者可以直接提供一支官方測試過的 Windows 字型安裝 PowerShell 腳本（或至少寫清楚：用 `fonts` 資料夾 + 登錄檔 `HKCU\...\Fonts`，每個字重要給不重複的 key），不要讓 AI/使用者自己土法煉鋼組合 COM API。
- 或考慮：字幕功能是否一定要「思源宋體全部 7 個字重」？如果只用得到 Regular/Bold 兩種，子集下載量可以從 132MB 降低很多，也減少犯錯機會。
- 132MB 的下載＋7 個字重的登錄檔寫入，在「大概要多久」的預期管理上應該讓使用者知道這步驟本身就要幾分鐘，不然中途容易以為卡住了。

## 三、Python 套件安裝階段（關鍵問題）

### 問題 4：`faster-whisper` 依賴的 `av`（PyAV）套件在 Windows 11 上被 Smart App Control 封鎖，這是這次安裝耗時最久、最難排查的一關

**現象**：`pip install faster-whisper` 本身「成功」（沒有報錯），但實際 `import faster_whisper` 時炸掉：
```
ImportError: DLL load failed while importing encparams: <亂碼，實際上是 cp950 主控台顯示 Unicode 訊息時的編碼錯誤>
```
在預設 Git Bash / 一般主控台編碼下，這行錯誤訊息會顯示成亂碼（因為系統原始訊息是中文，但主控台編碼是 cp950 而不是 UTF-8），第一時間完全看不出真正原因，很容易被誤判成「缺 DLL / 少裝 Visual C++ Redistributable」——這次也真的先照這個方向裝了 VC++ Redist，裝完問題依舊，才換成 PowerShell 用 UTF-8 編碼重新執行拿到完整訊息，才發現真正的字是：

> 應用程式控制原則已封鎖此檔案。

也就是 **Windows 11 的 Smart App Control（智慧型應用程式控制）** 判定 `faster-whisper` 透過 delvewheel 打包進 `av` 套件裡、檔名帶亂數雜湊的 FFmpeg DLL（例如 `avcodec-62-984de33...dll`）是「不受信任/未識別」的執行檔，直接擋下載入。

確認方式：
```powershell
Get-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\CI\Policy"
# VerifiedAndReputablePolicyState : 1  → Smart App Control 是開啟狀態
```

**這是一個安全性設定，AI 助手判斷不應該自己關掉**——Smart App Control 一旦關閉，必須重灌 Windows 才能重新開啟，屬於不可逆變更，應該由使用者自己決定，不能代為執行。目前只能請使用者自己選：
1. 自己去把 Smart App Control 關掉（`設定 → 隱私權與安全性 → Windows 安全性 → App 與瀏覽器控制 → Smart App Control`），風險是不可逆；
2. 改用 `openai-whisper`（透過 subprocess 呼叫外部 ffmpeg.exe，不會匯入未簽章 DLL，理論上能繞過封鎖，但要多裝 PyTorch，約 2-3GB，且 CPU 推論速度比 faster-whisper 慢很多）；
3. 先跳過逐字稿引擎，其他功能先能用。

這次使用者選了「先跳過」。

**建議（這塊影響最大，優先處理）**：
- 這是**全新 Windows 11 裝機的預設狀態**（Smart App Control 對一般消費機型是預設開啟的），代表任何一個沒重灌過電腦、直接開箱用的使用者都極可能撞到這個問題，不是個案。
- 強烈建議在 README「手動安裝」或「常見問題」區塊，明確寫出這個症狀（含「應用程式控制原則已封鎖此檔案」這句關鍵字，方便使用者/AI 直接搜尋比對）、原因（Smart App Control 擋 PyAV 的未簽章 DLL）、以及**不需要關閉系統安全性設定的替代方案**。
- 長期解法：評估幫 `faster-whisper`/`av` 換一個 Windows 上信譽較高、或有效簽章的發行管道；或是把 `openai-whisper` 直接做成 Windows 平台的預設/建議 backend（程式碼結構上 `transcribe.py` 已經是 mlx → faster-whisper 的 fallback 鏈，補一個 openai-whisper fallback 對開發者來說改動不大）。
- `setup.sh` 裝完 `faster-whisper` 後應該立刻嘗試 `import faster_whisper` 做 sanity check，失敗要當場印出清楚的中文提示（「偵測到 Windows 安全性原則封鎖，請見 README 疑難排解」），而不是留到使用者實際剪片時才爆炸。

### 問題 5：pip 在繁體中文 Windows 主控台下，`pip show` 這類指令本身會噴 `UnicodeEncodeError`

`pip show av` 印作者資訊時，作者名字裡有個 `é`（`Jeremy Lainé`），在 cp950（繁體中文 Windows 預設主控台編碼）底下印不出來，直接讓 pip 的 log handler 炸掉一串 traceback（雖然不影響安裝結果，但畫面上會噴出一大串看起來很嚇人的錯誤，容易讓使用者以為安裝失敗）。

**建議**：這個是 pip/rich 在繁中 Windows 環境的通病，開發者影響力有限，但可以在 README 提醒：若在終端機看到 `UnicodeEncodeError` / `cp950` 字樣，通常只是主控台編碼顯示問題，不代表安裝失敗，可忽略或改用 PowerShell 執行 `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8` 再重試。

## 五、給「小白使用者」的引導話術建議

這次全程是 AI 代裝，過程中有好幾個「要不要繼續」的判斷點。從使用者體驗角度，建議開發者可以在 README 或未來的 skill 指引裡，明確定義這幾個「該問使用者」的時機點跟建議話術，讓不同 AI 工具（Claude / Codex 等）代裝時體驗一致：

1. **系統層級安裝前，一次性列清楚要裝什麼、多大、要多久**，不要每個套件分開問。例如：
   > 「這台電腦缺 4 樣東西：Node.js、Python、ffmpeg、思源宋體字型，加起來大概要下載 300-400MB，安裝約 10-15 分鐘。都用官方來源（winget / GitHub 官方 release）自動裝，要開始嗎？」

   而不是每個工具跳一次確認，容易讓小白使用者覺得「怎麼一直跳出問題在問我」。

2. **遇到需要使用者自己判斷的風險決定（例如 Smart App Control）時，話術要避免直接丟技術名詞**，先講「這是什麼、為什麼要問你」，再給選項。例如可以引導成：
   > 「有一個 Windows 內建的安全防護擋住了一個安裝步驟。這個防護不建議關掉（一旦關掉要重灌電腦才能開回來），所以我不會自己關。你可以選：(A) 换一個不會被擋的替代方案，比較慢一點但安全 (B) 你自己決定要不要關閉這個防護 (C) 先跳過這步，其他功能一樣能用。」

   這次執行上大致符合這個精神（有先解釋不可逆風險、給選項），可以視為一個範例保留下來給開發者參考話術寫法。

3. **明確告知「現在這步可能要等 X 分鐘」**，尤其是思源宋體 132MB 下載跟 winget 安裝這種使用者只看得到終端機捲動、看不懂在幹嘛的步驟，要有進度提示的心理預期管理，不然容易讓小白使用者以為當機。

4. 建議 skill/README 明確規範：**遇到「pip install 成功但 import 失敗」這種情況，永遠要多做一步 `import` sanity check 再回報「裝好了」**，不要以 pip 的 exit code 為準——這次如果照 `setup.sh` 原本的流程走完，畫面會顯示「=== 完成 ===」，但其實逐字稿引擎完全不能用，使用者會在真正剪片時才發現，體驗上比在安裝當下就發現差很多。

## 六、替代方案實測：Purfview 的 whisper-standalone-win（Faster-Whisper-XXL）

因為問題 4（Smart App Control 封鎖 PyAV）目前無解，另外實測了一個社群方案：[Purfview/whisper-standalone-win](https://github.com/Purfview/whisper-standalone-win) 的 `Faster-Whisper-XXL`。這是把 faster-whisper 整個打包成單一 .exe（含所有函式庫，不需要 Python/pip），理論上可以繞過 pip 裝的 `av` 套件那個未簽章 DLL 問題。

**測試方式**：下載 `Faster-Whisper-XXL_r245.4_windows.7z`（官方 release，約 1.42GB），解壓縮後執行 `faster-whisper-xxl.exe`，用 Windows 內建語音合成（`System.Speech.Synthesis`，zh-TW 語音包「Microsoft Hanhan Desktop」）現生一段 28 秒繁體中文語音當測試音檔，跑 `--model medium --language zh`。

**結果**：

| 項目 | 結果 |
|---|---|
| 有沒有被 Smart App Control 擋 | **沒有**。exe 正常啟動、下載模型、推論、寫出結果檔，全程無攔截 |
| 首次執行耗時 | 8 分 59 秒（其中 8 分 30 秒是下載 medium 模型 1.53GB，之後重跑同模型不用再下載） |
| 純轉錄耗時（28 秒音檔） | 約 29 秒（CPU 推論，速度 1.29 audio-seconds/s，接近即時速度） |
| 中文辨識品質 | 佳。逐句比對只有一個同音字錯誤（「剪掉」→「简调」），其餘標點、斷句、時間軸都正確 |

**需要注意的差異**：
1. **輸出是簡體中文**，即使指定 `--language zh` 也一樣（「欢迎」「辨识」「时间轴」）。這個工具包字幕需求是繁體中文，若採用這條路線，pipeline 裡要**額外加一道簡轉繁**（例如 OpenCC），這是目前 `transcribe.py` 用 pip 版 faster-whisper 不會遇到的問題。
2. **體積明顯較大**：下載 1.42GB、解壓後約 4.4GB，比 pip 裝的 `faster-whisper`（連同 PyTorch/CTranslate2 等依賴通常數百 MB）大上不少，對「一鍵安裝」的使用者體驗有影響（下載時間、硬碟空間）。
3. **整合方式要改成 subprocess 呼叫外部 exe**，而不是現在 `transcribe.py` 裡用 `from faster_whisper import WhisperModel` 直接 import 的寫法，需要新增一個對應的 backend 分支（解析 exe 產出的 `.json`/`.srt`，轉成專案內部的標準 `{"words": [...]}` 格式）。

**建議**：可以把這個方案定位成「Windows 上遇到 Smart App Control 擋 PyAV 時的官方 fallback」，在 `transcribe.py` 新增一個 `whisper_xxl`（或類似命名）backend，偵測到 `av` import 失敗時提示使用者去下載這個 standalone exe 並指向下載路徑，同時內建簡轉繁的後處理，會比現在讓使用者卡在「先跳過逐字稿引擎」好很多。

## 七、追加測試：`windows-support` 分支的 `setup.ps1` / `whisper_xxl` 整合

在收到上面報告後，Jake 推了 `windows-support` 分支，新增 `setup.ps1`、`scripts/windows/install-font.ps1`，並在 `transcribe.py` 裡把 Purfview 的 Faster-Whisper-XXL 接成正式 `whisper_xxl` backend（含 OpenCC 簡轉繁）。以下是這個分支的實測結果。

### 7.1 `setup.ps1` 存成 UTF-8 沒有 BOM，在 Windows PowerShell 5.1 上直接解析失敗（最高優先）

照 README 指示的指令：
```
powershell -ExecutionPolicy Bypass -File .\setup.ps1
```
**連第一行 log 都沒印出來就崩潰**，錯誤是一連串語法錯誤（`Unexpected token '}'`、`The string is missing the terminator`）。

**病因**：`setup.ps1`（連同 `install-font.ps1`）是 UTF-8 編碼但**沒有 BOM**。這台機器只有內建的 **Windows PowerShell 5.1**（`$PSVersionTable.PSVersion` = 5.1.26100，`PSEdition` = Desktop，沒裝 PowerShell 7 / pwsh）。PowerShell 5.1 讀沒有 BOM 的 `.ps1` 檔時，是用**系統預設編碼**（這台是繁中 Big5/cp950）去解析，不是 UTF-8，於是檔案裡的中文字串全部被讀成亂碼，語法解析器跟著炸掉。

這**幾乎會擋住所有繁中 Windows 學員**——他們用的就是系統內建、預設不是 UTF-8 讀取模式的 Windows PowerShell 5.1，不是 pwsh。把檔案另存成 UTF-8 with BOM 後重跑，腳本邏輯本身完全沒問題、能順利跑完。

**建議**：把 `setup.ps1` 和 `scripts/windows/install-font.ps1` 都存成 **UTF-8 with BOM**（或至少加一行文件開頭偵測，提示使用者用 `Get-Content -Encoding UTF8 | Out-String | Invoke-Expression` 之類的方式繞開），這是目前唯一會讓「一鍵安裝」完全跑不動的問題，優先度應該最高。

### 7.2 字型安裝腳本沒問題

`install-font.ps1` 正確偵測到思源宋體已經裝過（掃使用者字型目錄），直接印 `[OK] 思源宋體已安裝(7 個檔案),略過`，冪等判斷邏輯正確，避開了第二節提到的兩個坑（重複複製、字重登錄檔互蓋）。

### 7.3 Smart App Control 的封鎖不是永久的——會自己解除，這點影響 fallback 設計

這是意外發現：**同一台機器、同一個 venv、完全沒改任何東西**，第一次測試時（第三節問題 4）pip 版 `faster-whisper`（`av` 套件）被 Smart App Control 擋下；隔了幾小時後在這次測試裡，**同一個 `av==18.0.0` 卻可以正常 `import` 了**。確認過 `HKLM:\SYSTEM\CurrentControlSet\Control\CI\Policy` 的 `VerifiedAndReputablePolicyState` 仍然是 `1`（Smart App Control 還是開著的），代表不是使用者關掉了防護，而是它背後的雲端信譽評分機制重新評估、判定這個檔案安全了，**不需要使用者做任何動作就自己解除封鎖**。

**這對 `setup.ps1` 目前的 fallback 邏輯有實際影響**：現在的做法是「pip 版一失敗，馬上引導下載 4.4GB 的 XXL 獨立版」，但既然封鎖有機率是暫時的，也許應該先提示使用者「过一段時間、或重開機後重跑一次 setup.ps1 再試」，不用每個人一撞到就得下載一大包。這點值得跟開發者討論怎麼取捨（多一次重試 vs. 使用者體驗的等待時間）。

### 7.4 簡轉繁只接在 `whisper_xxl` 分支，pip 版 faster-whisper 完全沒有轉換邏輯（功能缺口）

因為 8.3 提到的封鎖自己解除了，這次測試時 pip 版 `faster-whisper` 是可用的，`transcribe.py` 的自動判斷順序（mlx → faster-whisper(pip) → whisper_xxl）會優先選到 pip 版。用同一段測試音檔（跟第六節相同的 28 秒繁中語音），兩種方式輸出對比：

| 呼叫方式 | 輸出 |
|---|---|
| 不指定 backend（照 README 範例指令直接跑，會自動選到 pip 版） | ❌ 簡體：「欢迎使用这个人工智慧影片剪辑工具包」 |
| 明確指定 `--backend whisper_xxl` | ✅ 繁體：「歡迎使用這個人工智慧影片剪輯工具包」 |

原因：`_to_traditional()`（OpenCC `s2twp` 轉換）目前只寫在 `transcribe_whisper_xxl()` 裡，`transcribe_whisper()` 裡呼叫 pip 版 `faster-whisper` 的那段完全沒有接簡轉繁。而 pip 版 faster-whisper 對 `zh` 語言預設輸出的也是簡體（這是底層 Whisper 模型本身的傾向，不是 CTranslate2 實作差異），所以**只要 pip 版能跑（不管是原本就沒被擋，或像 8.3 那樣自己解除封鎖），使用者不管有沒有裝 XXL、不管有沒有指定 backend，都會拿到簡體字幕**，跟這個工具包「輸出繁體中文字幕」的核心目標矛盾。

**建議**：把 `_to_traditional()` 的呼叫從 `transcribe_whisper_xxl()` 提出來，變成 `transcribe_whisper()` 整個函式的共用後處理（不管走 mlx / faster-whisper / whisper_xxl 哪個分支，只要 `language` 是 `zh` 就統一跑一次簡轉繁），而不是只接在其中一個分支。這是這次測試裡除了 8.1 之外最重要的發現。

兩種情況辨識品質都很好，唯一的共同錯誤是同音字（「剪掉」被誤認成「簡調」），這是 Whisper 本身的辨識誤差，跟簡繁轉換無關。

### 7.5 winget 依賴檢查在重跑時誤判已安裝的套件

腳本用 `Get-Command <exe>` 判斷 Node.js / ffmpeg 是否已安裝，但這次重跑時（Node/ffmpeg 都已經裝過），腳本仍印出「安裝 Node.js LTS ...」「安裝 ffmpeg ...」試圖重裝。不確定是腳本邏輯本身的問題，還是這次測試環境的 PATH 快取殘留（`Refresh-Path` 只在整個迴圈跑完後呼叫一次，不是每裝完一個就更新，理論上不影響「已安裝判斷」，但無法完全排除環境差異）。另外腳本把 `winget install` 的輸出整個 `| Out-Null` 吞掉，看不出 winget 自己到底判定「已安裝、略過」還是真的重跑安裝流程——**建議至少保留 winget 自身的輸出，或檢查其 exit code**，方便之後除錯判斷是腳本問題還是環境問題。

### 7.6 UnicodeEncodeError / cp950 主控台亂碼問題依舊存在

跟第一次報告一樣，繁中 Windows 主控台印中文（`pip show`、直接輸出逐字稿文字）還是會噴 `UnicodeEncodeError: 'cp950' codec can't encode...`，不影響實際結果，但畫面上還是會嚇到人，維持原本報告的建議（提醒使用者這是顯示問題可忽略）。

## 八、總結：優先順序建議（更新版）

第一輪報告的建議（Smart App Control fallback、python3 空殼偵測、`setup.ps1`、字型腳本）都已經在 `windows-support` 分支實作，且第七節驗證邏輯基本正確，**只剩以下幾點還沒解決**：

1. **最高優先（新）**：`setup.ps1` / `install-font.ps1` 存成 UTF-8 沒有 BOM，在絕大多數學員會用的 Windows PowerShell 5.1（繁中系統）上**直接解析失敗、完全跑不起來**。這是目前唯一會讓整條「一鍵安裝」卡死在第一步的問題，必須先修（見 7.1）。
2. **最高優先（新）**：簡轉繁（OpenCC）目前只接在 `whisper_xxl` 分支，pip 版 `faster-whisper` 分支完全沒有轉換，只要 pip 版能跑（不管是原本沒被擋，還是像 7.3 那樣封鎖自己解除），使用者都會拿到簡體字幕。應該把簡轉繁後處理提到 `transcribe_whisper()` 共用邏輯（見 7.4）。
3. **中優先（新）**：Smart App Control 的封鎖被觀察到會自己解除（同一個檔案，隔幾小時就不再被擋），建議 fallback 邏輯考慮「先建議重跑一次再決定要不要下載 4.4GB 的 XXL」，而不是一失敗就要求下載一大包（見 7.3）。
4. **低優先（新）**：`setup.ps1` 重跑時對已安裝的 Node/ffmpeg 判斷可能不準，且 `winget install` 輸出被整個吞掉，不好排查是腳本問題還是環境問題（見 7.5）。
5. **低優先（延續）**：pip 在繁中主控台的 `UnicodeEncodeError` 提醒使用者可忽略，這次測試依然存在，開發者可視情況決定要不要處理。
