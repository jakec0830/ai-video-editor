# ai-edit — AI 剪輯流程

這份給 AI 讀。任何有終端機 + 檔案工具的 coding agent(Claude Code、Codex、Cursor)都能照著做。指令是純 shell,程式碼、路徑、技術名詞保留原文,說明用繁體中文。

**所有路徑相對於「工具包根目錄」**(有 `setup.sh`、`tools/`、`素材庫/`、`我的影片/` 的那層,以下叫 `KIT`)。深入技術參考見 `KIT/tools/PIPELINE-NOTES.md`。

**適用範圍:** 這個流程是給「口播影片」的(有人對著鏡頭講話)。剪接靠語音逐字稿,所以沒有講話的素材(純空景配樂那種)這個流程剪不了。底層的 HyperFrames 其實能做更多類型(產品片、解說片、動態圖文等),想延伸的話用 `npx hyperframes` 直接做 — 但那不走這個技能。

「問使用者」= 用你的介面能問就問;完全自動跑的話,選合理預設並在結尾說明。

## 開場:補 PATH、檢查更新、讀偏好檔(學習循環的起點)

每次開始前做三件事:

0. **`source KIT/tools/env.sh`**。它把 `~/.local/bin` 跟剛裝好的 Homebrew 補進這個 session 的 PATH,冪等、極快。沒做這步,家目錄安裝的工具會「明明裝好了卻 command not found」(實測:安裝時寫進 `~/.zshenv` 的 PATH,當下開著的 session 吃不到 — 環境是 session 開始時快照的)。順帶看一眼 `KIT/錯誤回報/`:有沒有檔名**不是** `已送-` 開頭的報告(例如上一個 session 留下的安裝紀錄、送失敗的復盤)?有的話問使用者一句「上次有份報告還沒傳給 Jake,現在幫你補送?」,同意就送(送成功後把檔名加上 `已送-` 前綴,下次就不會重複問)。

1. **檢查有沒有新版本**(只有當 `KIT/.git` 存在,也就是用 git clone 下載的才做)。跑 `git -C KIT fetch --quiet`,再看 `git -C KIT rev-list HEAD..@{u} --count`。結果 > 0 代表 Jake 出了新版 → 問使用者「有新版本,幫你更新嗎?」,說好就跑 `bash KIT/tools/update.sh`(等同 `git -C KIT pull`)。**更新完一定接著跑一次 `bash KIT/setup.sh`(Windows 用 `powershell -ExecutionPolicy Bypass -File KIT\setup.ps1`)。** `update.sh` 只拉新檔案,不會重跑安裝檢查 — 如果這次更新動到套件依賴(例如新增 opencc、換了 Whisper 後端),使用者環境不會自動跟上,一定要重跑 setup 才會裝到新套件 / 重新驗證。setup 腳本本身是冪等的(已經裝過的東西會跳過),重跑不會壞任何東西,所以**每次更新後都跑,不用另外判斷這次更新是不是牽涉依賴**。使用者的影片、素材、偏好都不會被動到(那些沒進 git)。如果不是 git 資料夾(ZIP 下載的),**跳過這步,不要報錯**。
2. **讀 `KIT/我的剪輯偏好.md`**。裡面是這位使用者過去累積的習慣(字幕風格、愛用音樂、剪接節奏等)。照著走,不要每次重問已經講過的偏好。收尾時你會更新它(見文末)。若這個檔不存在(第一次跑),從 `KIT/tools/我的剪輯偏好.範本.md` 複製一份(`setup.sh` 會自動做)。

## 第一次設定 — 一次講清楚要裝什麼,再帶著裝

**大原則:使用者不開終端機。** 所有指令都是你跑;交給使用者的只有三種動作:在 Claude Code 按「允許」、在 Mac 自己跳出的視窗點按鈕或輸密碼、在瀏覽器登入。開始前先用一句話講清楚:「等下所有指令我來跑,你不用開終端機、不用打指令,偶爾按個允許或在 Mac 的視窗點一下就好。密碼只會打在 Mac 自己跳出的視窗,永遠不要打在聊天視窗給我。」把過程中發生的事(裝了什麼、卡在哪、錯誤訊息)記到 `KIT/錯誤回報/安裝紀錄-<YYYY-MM-DD>.md`,裝完照下面第 6 步處理。

使用者說「幫我安裝」「幫我設定」或這是第一次跑時,主動幫他把缺的東西裝好。學員是新手。這段的話術原則來自真實裝機記錄(Mac 跟 Windows 都有),照做體驗差很多:

1. **先跑檢查、彙總、一次確認**(不要每個套件分開問,新手會覺得「怎麼一直跳問題」):
   - Mac/Linux 跑 `bash setup.sh`、Windows 跑 `powershell -ExecutionPolicy Bypass -File setup.ps1`,**看 exit code**:0 = 就緒;非 0 時結尾會列「還沒完成 — 缺:…」,照那份清單補。
   - 把所有缺的東西**彙總成一句**跟使用者確認,含大小跟時間預期,例:「這台電腦缺 3 樣:Node、ffmpeg、思源宋體字型,加起來約 300-400MB、裝 10-15 分鐘,都用官方來源自動裝,要開始嗎?」
   - **要等的步驟先講要等多久**(字型 132MB、winget 下載、XXL 1.4GB 這種),不然使用者會以為當機。
2. **Mac 沒有 Homebrew 時(全新 Mac 很常見)— 走官方 .pkg 圖形安裝,不碰終端機:**
   1. 一句話說明:「你的 Mac 缺一個叫 Homebrew 的工具管理員(官方、免費),裝好以後我裝其他東西都不再需要密碼。我會下載官方安裝檔(約 130MB),你像裝一般軟體一樣點幾下就好。」
   2. 從 `https://github.com/Homebrew/brew/releases/latest` 下載官方 `Homebrew.pkg`,下載完 `open` 它讓安裝視窗跳出來。
   3. 帶著點:繼續 → 安裝 → 這時 Mac 會用**它自己的視窗**要密碼。話術照講:「這是 macOS 在問,不是我。是你**重開機登入 Mac 打的那組密碼**,不是 Apple ID 的(平常都用 Touch ID 解鎖的人也有這組)。打在那個視窗裡,不要打在聊天視窗。」 → 完成。
   4. 用完整路徑驗證 `/opt/homebrew/bin/brew --version`(新裝的 brew 不在這個 session 的 PATH — 重新 `source KIT/tools/env.sh` 就有了),把 `eval "$(/opt/homebrew/bin/brew shellenv)"` 寫進 `~/.zprofile`(你自己寫檔,使用者不用動手),然後**把下載的 Homebrew.pkg 刪掉**(129MB,用完即丟)。
   5. 密碼想不起來、或使用者不想輸入 → 不要卡關也不要勸,直接走第 3 步的免密碼備案,一樣裝得起來。
3. **免密碼備案(全部裝進家目錄,永遠不需要密碼)** — 只在 .pkg 那條走不通時用:
   - Node:nodejs.org 官方 tarball → 解到 `~/.local/opt/node`,symlink 進 `~/.local/bin`。
   - 思源宋體:Adobe 官方 GitHub releases 的 `SourceHanSerif-VF.otf.ttc` → `~/Library/Fonts/`。
   - ffmpeg/ffprobe:社群 static build(下載後**驗它公布的 SHA256**)→ `~/.local/bin`,`xattr -c` 清隔離屬性。先用白話取得同意:「影片引擎這份是社群編譯版 — Mac 沒有官方版,這是不輸密碼的唯一辦法。畫質跟官方完全一樣,差別只在來源信任度,OK 嗎?」
   - PATH 寫進 **`~/.zshenv`,不是 `.zshrc`**(非互動 shell 讀不到 `.zshrc`,setup 會誤判成沒裝);寫完**這個 session 還是要 `source KIT/tools/env.sh`** — `.zshenv` 只對之後新開的 shell 生效。
4. 裝完**重跑一次 setup 腳本**,確認 exit code 是 0。腳本已涵蓋各平台的安裝(Mac brew / Windows winget / 字型腳本),不用自己拼指令;腳本搞不定的例外才照 README 手動裝。
5. **「pip 裝成功」不等於「能用」** — 任何用 pip/npm 裝的關鍵套件,回報「裝好了」之前要實際 import/執行一次驗證(setup 腳本對 whisper 引擎已內建這個檢查;你自己額外裝東西時也守這條)。實測教訓:Windows 上 faster-whisper pip 顯示成功、import 才爆,如果只看 pip 結果,使用者會在剪片剪到一半才發現。
6. **收安裝紀錄**:setup exit 0 之後,把過程紀錄(`錯誤回報/安裝紀錄-<日期>.md`)補完(結果、裝了什麼、卡過哪裡),問一句要不要傳給 Jake(流程同文末「出錯回報」;一切順利的紀錄也值得傳 — Jake 要知道哪些機器一次過)。

**heygen(線上音效庫)不屬於第一次安裝。** 它是選配,`素材庫/預設包/` 的音效跟 BGM 夠第一天用。只有使用者主動要更多音效/音樂時才裝,流程見文末「heygen 設定(選配)」。

**這些交給使用者做(你只能引導,不能代做):** 在 Mac 自己跳出的視窗輸入密碼、點圖形安裝視窗的按鈕、在瀏覽器登入。清楚說「這步換你做:要做什麼、為什麼」,等他做完再繼續。**絕不請使用者開終端機或自己打指令** — 需要指令的一律你來跑,被權限機制擋下就請他按「允許」(先講清楚那個指令在做什麼)。

**碰到安全性防護擋路時(例:Windows Smart App Control 擋 faster-whisper)**:絕不代替使用者關閉系統安全設定 — Smart App Control 一關就要重灌才開得回來。話術先講人話再給選項,不要丟術語:「有個 Windows 內建的安全防護擋住一步。這個防護不建議關(關了要重灌才能開回來),所以我不會動它。你可以選:(A) 等幾小時或重開機再試一次,這個封鎖常會自己解除 (B) 換一個不會被擋的替代方案(要多下載 1.4GB) (C) 先跳過,其他功能照常用。」

**底線:** Claude Code 本身的安裝跟登入是在這個技能能跑之前就完成的(不然使用者沒辦法跟你講話)。永遠不要嘗試從這裡安裝 Claude Code。

**轉逐字稿是跨平台的。** `transcribe.py` 會自動判斷後端:Apple Silicon 用 `mlx-whisper`(最快),Intel Mac／Linux 用 `faster-whisper`,Windows 用 `faster-whisper-xxl` 獨立版(pip 版 faster-whisper 在 Windows 會被 Smart App Control 擋,獨立 exe 不會)。同樣的模型、同樣會有重複字出錯的問題,所以下面的 xref／波形驗證步驟三邊都一樣重要。Windows 的 XXL 產出是簡體,`transcribe.py` 已內建 OpenCC 自動轉繁,不用你處理。若 Windows 使用者的逐字稿引擎沒就緒(exe 沒放好),`transcribe.py` 會直接報錯並指到下載位置 — 照 README「Windows 疑難排解」引導他放好 `tools/whisper-xxl/`。Windows 的一鍵安裝是 `setup.ps1`(PowerShell),不是 setup.sh。

設定好之後,下面簡寫:`PY="$KIT/tools/freecut/.venv/bin/python3"`、`H="$KIT/tools/freecut/helpers"`。

## 資料夾結構與清理 — 幫新手保持整齊

**一支影片一個資料夾**,放在 `KIT/我的影片/` 底下。使用者開始時,建好這個結構並用一句話跟他說明:

```
KIT/我的影片/<專案名>/
  <原始影片檔>        原檔,絕不動它
  成品.mp4            完成的影片 — 他要的就這一個
  審片區/             使用者唯一要開的資料夾:要審的影片 + 字幕.json,兩個檔一起拖進 審片.html
  工作檔/             其他全部丟這(使用者平常不用開):
    transcripts/       逐字稿 JSON + <名稱>.words.txt(compact,讀這份)
    edl.json           剪接配方(唯一真相)
    preview_v*.mp4     預覽(舊的邊做邊刪,只留最新)
    captions/          字幕專案(captions.json + index.html)
    專案筆記.md        這支的錯字表/選曲/文案/進度(隔天回來先讀,範本在 tools/freecut/templates/)
```

**審片區 = 給使用者的窗口。** 每次要請使用者審,就把「當前 preview + 字幕」複製進去(見第 4 步),裡面永遠只有這兩個檔 — 他不用進 工作檔/ 翻。舊版本直接覆蓋,不留歷史。

**配方 vs 媒體**:`transcripts/`、`edl.json`、`captions/*.json`、`專案筆記.md` 是「配方」(文字、極小、隔天要用,cleanup 保留);preview、render、擷取圖、node_modules、審片區的影片是「媒體」(大、可重生,cleanup 刪)。**例外:專案完成後,審片區裡的最終版(影片 + 字幕.json)保留不刪** — 那是使用者重看成品的窗口(cleanup.sh 已內建這個例外)。

紀律(照做,他的硬碟才不會爆):
- 每個中間檔都放 `工作檔/` 底下。絕不把 preview、render、擷取的 PNG、試聽的音檔散在最上層或旁邊的資料夾。
- 舊的 preview 邊做邊刪 — 有了 `preview_v2` 就刪掉 `preview_v1`。只留最新的 preview + `edl.json`。
- 擷取的畫面 PNG、試聽下載的音檔是用完即丟:看完馬上刪。
- **一次工作結束時,主動問要不要清理:**「要不要幫你清一下工作檔?我會刪掉大媒體檔(preview、擷取的圖、node_modules),但**留下配方文字檔**(edl.json、逐字稿、captions、專案筆記),下次回來重跑 render 就好,不用重轉稿重對時間。」顯示大小,等他說好。
- 使用者說「清理」「太亂了」時,跑 `bash tools/cleanup.sh <專案資料夾>`(安全 — 先顯示要刪什麼並要你確認;**只刪大媒體檔 + 可重生資料夾,保留所有文字配方**)。**絕不刪原始影片或成品.mp4。** 重點:配方檔(edl.json、transcripts/、captions/*.json/*.html、專案筆記.md)加起來通常 <1MB,卻是隔天回來的省時關鍵 — 別跟著媒體一起刪。

## 素材庫與 b-roll — 建議循環

`KIT/素材庫/` 是跨專案共用的可重複素材:`背景音樂/`、`音效/`、`b-roll/`、`圖片/`。詳見 `KIT/素材庫/說明.md`。

- 剪片要用音樂／音效／圖片／b-roll 時,**先看素材庫有沒有現成的**,有就直接用,不要重新下載。
- 使用者這次帶進來或下載了新的音樂／音效／圖片/b-roll,用完後**主動問**:「要不要把這個存進素材庫,下次可以重複用?」說要就幫他放進對的子資料夾。
- **b-roll**(補充畫面)放 `素材庫/b-roll/`:空景、產品特寫、示範操作畫面,任何口播中間想切過去的畫面。剪片時遇到適合插 b-roll 的段落(例如講到某個具體東西),**主動提醒**:「這裡可以插一段畫面,你素材庫裡有沒有相關的?或想拍一段?」這是**建議**,放哪、放不放由使用者決定,不要自動塞。實際怎麼合成見流程「5.5 插 b-roll(選配)」。

## 硬性工作順序 — 先鎖定「剪對了」再花錢

這是流程唯一的硬規則。它只管「順序」跟「檢查」,不管風格、節奏、特效(那些永遠是活的、量身的)。之所以要硬,是因為實測踩過:字幕/剪接的錯留到合成之後才發現,每修一次就是一次 3 分鐘 render + 重混音;而同樣的錯在「還是文字」的階段修,是 5 秒的事。原則:**上游沒鎖,絕不動下游。**

```
A. 轉逐字稿 → 一次把所有 xref 旗標用 split_blobs 查完(不要邊被嫌邊查)
B. 結構剪接 → 寫 edl.json → render preview(--preview,無字幕)
C. 機器對稿:verify_cut.py 比對 preview 跟「應該講的話」,先幫你抓出接縫 bug
      (夾字、漏字、兩句相撞)——這關過了,使用者才會看到版本
D. 出字幕(先當文字,還沒燒進畫面):gen_captions 產 captions.json
E. 使用者用「preview + 字幕檔」在 審片.html 看:一次收「剪接 + 字幕」的回饋
      (剪接類/字幕錯字/斷句,全部這輪一起改,都還很便宜)
F. 鎖定 → 剪接跟字幕文字從此不再動
G. 才開始:節奏提案 → 特效/運鏡/b-roll/吉祥物角色 → 合成 render
H. 音效 + 背景音樂混音 → 成品.mp4
```

**文字是地圖,不是裁判。** 使用者永遠用「有畫面的版本」下判斷(眼睛看死白/搖頭空白/醜接點、耳朵聽節奏),文字只是拿來精準指位置。所以 C 之後一定給 preview,不是只給一段文字要他核准。preview 很便宜(1 分鐘內),沒有理由拖到最後才給畫面。

**用字幕句指位置,不要用秒數。** 秒數每重剪一次就位移,講「那句『那中英的』不對」比講「1:04」穩,而且 審片.html 可以直接點那句跳過去。

## 剪輯流程

在 `<專案>/工作檔/` 裡作業(就是下面的 `--edit-dir`)。最後成品輸出到 `<專案>/成品.mp4`。照上面的「硬性工作順序」跑。

### 1. 轉逐字稿 + 交叉比對(剪之前先做)

```bash
$PY $H/transcribe.py <影片> --backend whisper --language zh --edit-dir <專案>/工作檔
$PY $H/xref_silence.py <影片> <專案>/工作檔/transcripts/<名稱>.json
```

- **讀逐字稿讀 `transcripts/<名稱>.words.txt`(compact 版),不要讀那份肥 JSON。** transcribe 會順手產這份一行一字的檔(`idx  start-end  字`),token 只有 JSON 的約 1/6。剪接、對時間、規劃字幕要的都在這份。
- **一次把 xref 的所有旗標查完,不要邊被使用者嫌邊查。** 每個 MERGE / LONG 旗標跑 `$PY $H/split_blobs.py <影片> <start> <end>`,在提剪接案之前就把真正的語音區塊看清楚。MERGE 區是 Whisper 出錯的地方(重複字被併成一個 token、贅字被吃進字的時間裡);實測**幾乎每個接縫 bug 都落在這些旗標區**。絕不在旗標區只憑逐字稿時間做細剪。這關先做完,後面才不會來回。
- **不要讀 `hyperframes init` 產生的 `transcript.json`。** 它會用自己的英文小模型把中文影片重轉一份,整份是幻覺(實測整段「I think it is important…」重複幾十遍,燒掉幾萬 token)。字幕一律以 `工作檔/transcripts/` 這份為準。

### 2. 結構剪接(第一輪 — 依逐字稿)

先用白話提案:哪個 take 留、哪些 NG／重拍刪、哪些停頓修短。**句間留白不要整支固定同一個值** — 短影音從約 250ms 起跳,要衝節奏就收到 150ms,關鍵句／轉折前放到 400ms 以上讓它喘一口。太緊(130ms 以下)會很躁,太鬆(400ms+ 全程)會拖。等使用者確認,再寫 `工作檔/edl.json`:

```json
{"version":1,"sources":{"NAME":"/abs/path.MOV"},
 "ranges":[{"source":"NAME","start":1.53,"end":7.54,"beat":"HOOK","quote":"...","reason":"..."}],
 "grade":"none","overlays":[],"subtitles":null,"total_duration_s":0}
```

range 的時間全部用**原始影片的秒數**。輸出:`$PY $H/render.py 工作檔/edl.json -o 工作檔/preview_v1.mp4 --preview --no-subtitles`

### 3. 細剪(第二輪 — 使用者主導,用波形當真相)

使用者用耳朵聽 preview。任何要修的地方:
- 跑 `$PY $H/timeline_view.py <原始影片> <start> <end> --n-frames 20 -o wave.png` 給他看 — 使用者從波形的時間軸讀出精確秒數。
- 修正一律用**原始影片時間**(重剪也不會跑掉)。輸出時間軸的秒數每次重剪都會位移 — 絕不拿舊數字去加減offset。
- 你聽不到聲音。字到底是什麼字,由使用者的耳朵或波形決定,不要重跑 Whisper(換大模型不會解決重複字問題)。

**每次 render 完 preview,先自己做兩件事,再拿給使用者:**

1. **機器對稿(接縫檢查)** — 這關幫你把接縫 bug 擋在使用者看到之前:
   ```bash
   $PY $H/verify_cut.py <專案>/工作檔/preview_vN.mp4 \
       <專案>/工作檔/transcripts/<名稱>.json <專案>/工作檔/edl.json --fixes fixes.json
   ```
   它重聽 preview、跟「EDL 該講的話」比對,只回報**結構性**的差異(某字被夾掉、某段漏進來、兩句相撞),Whisper 的小誤聽會自動抵消。有旗標的點,用波形/擷取影格確認再修。這關過了(或旗標都確認過)才給使用者看。
2. **整句讀回** — 別只在心裡想接縫。把剪出來的句子當**連續腳本**讀一遍(可直接看 verify_cut 的比對,或重轉 preview 讀 words.txt),確認沒有半句話、沒有殘留的重複、沒有主詞相撞(例:剪掉中間後「我自己」直接撞「我大概」會不通順)。這個檢查花 10 秒、零張圖,是必做,不是等使用者來抓。

**不要幫沒聽清楚的字瞎猜。** 錯字修正只改「明確聽得出來」的字;拿不準的字(尤其 MERGE 區、中英夾雜)一次列給使用者確認,不要自己編一個(實測「總而言之」被瞎猜成「那中英的」,連錯字字典都跟著錯)。

### 3.5 節奏提案(這步是靈魂)

> 順序上這是「硬性工作順序」的 **G**:等剪接跟字幕文字都在 審片 那輪鎖定之後才做,不是剪完馬上做。放在這裡是因為它接著第 5/5.5 步的特效執行。鎖定前不要開始堆特效。

新手不會自己判斷哪裡該加什麼。**由你(AI)讀逐字稿,對這支影片提一份具體的節奏提案**,預設值先幫他填好,讓他說「好」就有 80 分,想改再改。工具箱是你的調色盤,不是丟給他自己選。

鉤子先講(前 3 秒決定有沒有人看下去,最需要引導):

**A. 鉤子 = 口語鉤子 + 視覺鉤子**
- **口語鉤子**:他錄的第一句 → 剪緊、不留死白、快速落地(放大他錄的,不是幫他選鉤子)。
- **視覺鉤子**(重點,多花力氣引導):讀懂第一句在講什麼,主動說「你的鉤子在講 X,前 3 秒放一個 X 的畫面」。
  - 有相關圖片／b-roll → 用他的。沒有 → 你直接做一張醒目的**文字／圖文卡**把關鍵句變成畫面(不用素材,新手也一定有視覺鉤子)。
  - 課程核心原則:第一句一定要「把講的東西用畫面呈現出來」,加了視覺就放大。

**B. 其餘節奏工具(預設力度抓 75 分,有用量上限)**

預設提案的力度目標是 **75 分**,不是保守的 60、也不是全上 100。為什麼:100 分 = 每個 beat 都設計(像 demo 片),會多好幾輪 render 跟幾萬 token;75 分 = 挑最有力的點做滿,有記憶點、成本合理。使用者說「做滿一點」再加,說「簡單就好」再減。以 60-90 秒口播為基準,預設量:
- 字幕塗黃 3-4 個 · 運鏡(punch-in / step-zoom 合計)2-3 處 · 圖文卡 1-2 張
- b-roll 2-3 段(有素材才排) · SFX 2-4 個點 + BGM 全程 · 彩蛋類(縮框吐槽/吉祥物)0-1 個

配音效跟運鏡時,**照 `素材庫/預設包/對照表.md` 的「情緒/字眼 → 音效+運鏡」準則配**(riser 鋪在揭曉前、pop 配列點、punch-in 配強調句…素材就在預設包裡)。把它當工具箱:挑對的、在對的點、不疊在同一拍。各工具的用法:
- **字幕關鍵字 highlight**:整支挑 3-4 個最有力的詞塗黃,不是每句;已經有 b-roll／推鏡的那個點就不要再疊。**使用者在審片時反白/【詞】自己標過的優先算進額度**,你只補提案剩下的名額(他標太多就提醒超過上限,一起挑掉)。
- **切走 cutaway**:講到現場沒拍到／需要看到的具體東西 → 短暫全屏切過去,人聲不斷。要他的素材,沒有就跳過(不卡)。
- **畫中畫 PiP**:靈活用(表情／反應重要時保留臉,或任何時候覺得有用)。
- **插圖文卡**:講到看不見的抽象東西(地點/數據/品牌)→ 一張卡 ~1 秒。
- **運鏡動態**(讓畫面活起來,已驗證,做法見 5.1):
  - **推鏡 punch-in／拉遠 punch-out**:同一段口播畫面放大／縮小,像換第二機位或強調某句。
  - **平移 pan**:畫面左右緩緩移。
  - **靜態圖 Ken Burns**:插進來的靜態圖片加緩慢縮放+平移,不要死板一張。
  - **縮框(吐槽用)**:講到梗／反差時,整個畫面**瞬間**縮小露出黑邊 — 瞬切不是平滑動畫,突然才好笑。

**提案格式**:列一份簡短、可勾選的清單(時間點 + 做什麼 + 為什麼),預設值先幫他填好。使用者說「好」就照做;要改就改某一條。需要他素材的(cutaway／圖片)問一句「有沒有?沒有就跳過」——**絕不因為缺素材卡住**,其餘照樣到 80 分。這份提案接著就是第 4/5/5.5 步的執行清單。

**防套版**:提案是針對「這支的腳本、他的鉤子詞、他強調的地方」量身出的,不是固定模板;加上用量上限,天生就克制。不要出現機械式 pattern(不要「永遠第 3 個字塗黃」「永遠 2 秒切」)。

### 4. 字幕(硬性順序的 D — 先當文字,還沒特效)

字幕在**剪接鎖定前**就要出,而且**先當文字檔給使用者核**(斷句、錯字、highlight 都是文字階段的 5 秒修改)。順序:先產 captions.json → gen_captions 產乾淨的 index.html → 使用者在 審片.html 看 preview + 這份字幕。

**第一步:captions.json(時間 + 斷句 + highlight)**
```bash
$PY $H/edl_to_captions.py <transcript.json> <edl.json> -o captions.json --fixes fixes.json
```
它做 EDL 重算、英文碎片合併、時間夾緊、斷句分組。跑完你只做判斷層:修錯字(只改聽得清楚的)、調斷句、挑 highlight。**上字幕前把斷句清單當文字給使用者看**,讓他重斷。改斷句 / highlight 就在 captions.json 裡加 `"hl":"某詞"`(整句期間塗黃,靜態)。每次剪接改動後從頭重跑,絕不拿舊數字加減。

**第二步:gen_captions 產 index.html(不要手寫)**
```bash
mkdir -p <專案>/工作檔/captions && cd 進去
npx hyperframes init . --non-interactive --video ../preview_vN.mp4   # 只為了拿 npm 專案骨架
$PY $H/gen_captions.py ../captions.json --video ../preview_vN.mp4 \
    --w 1080 --h 1920 --duration <ffprobe量到的秒數> --font 宋體 -o index.html
```
`gen_captions` 把「工程上會出錯的部分」全部封好了(這些坑手寫必踩):影片 `data-start="0"`、字幕 `z-index` 要壓過影片不然字幕跑到影片後面看不到、每個 clip 要 `class="clip"`+`data-*`+唯一 id、`@font-face` 不寫字型會 fallback 成黑體。它留了 `CREATIVE LAYER` / `CREATIVE TIMELINE` 兩個插槽給你加特效(第 5/5.5 步)。**框架穩、內容活** — 斷句/highlight/特效都還是你量身出的。

字型:**兩個預設用中文名讓使用者選**(第一次問,記進偏好檔):**宋體**(襯線、文青,`--font 宋體`)/ **黑體**(無襯線、有力、短影音常見,`--font 黑體`)。**highlight 只換顏色(黃),絕不換字型。**

其他鐵則:句子層級字幕、連續講話不留空檔;**不做逐字卡拉OK highlight**(Whisper 逐字時間不夠準,量產會出包);`data-duration` 用 ffprobe 實際量到的長度不用 EDL 加總(loudnorm 會多約 0.2 秒)。

改完一定 `npx hyperframes lint` → 修掉 error → render。

**第三步:整備審片區,使用者在 審片.html 核(硬性順序的 E)**

先把「要審的兩個檔」放進專案的 `審片區/`(使用者唯一要開的資料夾,舊的直接覆蓋):
```bash
mkdir -p "<專案>/審片區"
cp "<專案>/工作檔/preview_vN.mp4" "<專案>/審片區/"
cp "<專案>/工作檔/captions/captions.json" "<專案>/審片區/字幕.json"
```
然後告訴使用者:
> 打開工具包根目錄的 `審片.html`,把專案裡「審片區」資料夾的**兩個檔一起**拖進去(影片 + 字幕.json)就能開始審。

他看得到字幕句、點句子跳位、雙擊改錯字/打「/」斷句/併下句/用【詞】標塗黃(影片上即時看到黃字效果)、拖區間剪掉幾秒到幾秒 — 全部動作都會變成註解,最後一次「複製全部」貼回來。這輪一次收「剪接 + 字幕錯字 + 斷句」全部的回饋,一起改,都還便宜。收斂後才鎖定,進第 5 步特效。**每出新版 preview,重跑上面的 cp 覆蓋審片區。**

### 5. 特效(選配)

GSAP 疊層,跟字幕同一個 composition:
- 對手勢／動作的特效,「放哪」「什麼時候」是**視覺問題**:擷取影格(`ffmpeg -ss T -i vid -vframes 1 f.png`)去看,把元素放在手／動作的位置。逐字稿沒辦法告訴你手在哪、手指什麼時候彈。
- 用真的品牌素材,不要自己畫(catalog 沒有的使用者自己給 — 資料庫有缺是正常的)
- 只用有限次重複、cubic easing,重疊的 tween 加 `overwrite:"auto"`

### 5.1 運鏡動態(推鏡／平移／Ken Burns／縮框)— 做法與驗證過的坑

動態全部是**用 GSAP 動 `transform`(scale / x / y)**在同一個 composition,不重壓影片。共通做法 + 踩過的坑:

- **推鏡 / 拉遠(scale)**:`transform-origin` 放臉的位置(直式約 `50% 42%`),放大幅度跟焦點是**視覺問題** — 擷取影格看臉在哪、留多少空間再決定幅度(不是固定值,參考值 1.1–1.2)。`tl.to("#vid",{scale:1.16,...})` 進、再 `to({scale:1.0})` 回。
- **平移(pan)**:先放大留邊,**縮放幅度一定要大於平移量,不然會露黑邊**(例:scale 1.2 → 每邊約 108px 邊界,平移 ±90 才安全又看得到)。**不要用 `fromTo` 設起始值** — `fromTo` 的 `immediateRender` 會把起始位移套到整段開頭(畫面一開始就歪、露黑邊)。改用「在平移那一刻 `tl.set(起始值)` + `tl.to(結束值)`」。
- **Ken Burns**:對插進來的 `<img>` 做緩慢 `scale`+`x` 位移(ease `"none"`),讓靜態圖有生命。
- **縮框(吐槽)**:用 `tl.set("#vid",{scale:0.78})` **瞬間**縮、再 `tl.set({scale:1.0})` 瞬間回 — **不要 `to`/不要 duration**,突兀才是笑點。
- **每加一個動態,一定擷取影格自己看過**(起點、中段、終點):有沒有露黑邊?看不看得出來?太快／太慢?— 這步不能省(平移的黑邊跟看不見就是漏了這步才發生的)。

### 5.5 插 b-roll(選配)

補充畫面。放哪由使用者／建議決定(見上面「素材庫與 b-roll」),不自動放。跟字幕特效同一個 composition,b-roll 就是**在主影片上面多疊一個 `<video>`**,主影片的人聲繼續播(音訊是獨立的 track)。兩種模式:

- **整段切換(cutaway)**:b-roll 蓋滿整個畫面(full canvas, `z-index` 蓋過主影片),在那幾秒把人臉整個換掉,聲音照播。
- **畫中畫(PiP)**:b-roll 縮成角落小窗(定位 + `border-radius` + 陰影),人臉留在後面。

驗證過的三個坑(踩到 render 會出錯或畫面凍住):
1. b-roll 的 `<video>` 要**自己當一個 clip**(直接掛 `data-start`/`data-duration`/`class="clip"`),**不要包在另一個有 `data-start` 的 `<div>` 裡** — 巢狀 render 會把它凍住。樣式(圓角、陰影、full/角落)直接寫在 `<video>` 上。
2. 每個 `<video>` 要有**唯一 id**,不然 render 找不到會凍住。
3. `data-track-index` 是**時間軌不是圖層**:host 跟它的媒體如果時間重疊,就不能同一個 track;前後圖層用 CSS `z-index` 控。想切片段用 `data-media-start` 挑素材的起點。

改完照樣 `npx hyperframes lint` → 修 error → render。

### 6. 輸出來看

```bash
npx hyperframes render --quality standard --output <專案>/工作檔/render_final.mp4
```

**render 一律輸出到 `工作檔/render_final.mp4`,不要直接輸出成 `成品.mp4`** — 成品.mp4 只能由第 7 步的混音(或無音效時的複製)產生,不然「render 版」跟「混音版」會變成兩個成品打架(實測學員回報過:專案裡出現兩份成品、還多一個成品資料夾)。規則:**成品.mp4 永遠只有一個、永遠在專案根目錄、絕不建「成品/」資料夾**;產出新成品前先刪舊的。

`--quality draft` 輸出的 codec 只能給 AI 自己擷取影格檢查用 — 在播放器裡會播不完整。**任何要給使用者看的一定要 `standard`。** render 完用 ffprobe 確認長度;**擷取影格檢查用 `ffmpeg tile` 把多個檢查點拼成一張圖**(例:`-filter_complex "[0][1][2]...hstack"`),AI 讀 1 張代替 6 張(~1.5k vs ~9k token),再跟使用者說做好了。

**render 可能很久,尤其 4K 素材(preview 也可能 5–10 分鐘)。** 一開始就背景跑、等它真的結束再往下,不要用前景預設 timeout(會白等一輪):Claude Code 用 `run_in_background`,或 `nohup … &` 然後等實際的 ffmpeg/ render 程序結束(wrapper 指令退出 ≠ 轉檔完成 — 實測因此誤判過)。用 ffprobe 確認輸出檔長度對了才算完成。**磁碟要留空間**(render 的暫存檔很大);空間不足會靜默失敗,先清 `$TMPDIR/hyperframes-*` 快取。

### 6.5 審片回饋(批次修改)— 用審片工具收註解,一次改完

出了 preview 給使用者看之後,與其讓他一句一句回「0:12 這裡…」「0:34 那裡…」(每次來回你都要重讀一次 context,很燒 token 又慢),請引導他用**審片工具**把所有要改的地方一次收齊,再一次貼回來。

告訴使用者:
> 你可以用審片工具一次記下所有要改的地方。打開工具包根目錄的 `審片.html`(雙擊用瀏覽器開),把專案裡「審片區」資料夾的**兩個檔一起拖進去**(影片 + 字幕.json)。右邊會出現字幕句:點句跳位、雙擊改錯字、打「/」斷句、「併下句」合併、拖區間條選幾秒到幾秒剪掉。全部標完按「複製全部」貼回來,我一次改完。

(記得先照第 4 步把最新的 preview + 字幕.json 複製進 `審片區/`。)

載入字幕檔後,註解會**錨在字幕句上**,貼回來長這樣(這種比純秒數穩,重剪也不跑掉):
```
影片審片註解
共 5 則,依時間排序

「開頭那我這是第二支」(0:04) → 太鬆剪掉一點
「這個出現的雛形」(0:12) → 字幕蓋到臉,往上移
[0:34] 推進 zoom in 在她說重點時        ← 沒錨到句子的還是純秒數
...
```
有「」的用那句字幕當座標(在 words.txt 裡搜那句話定位,不要用秒數);沒錨到的才退回秒數。

收到就**一次全部處理,不要一則一則來回**:
1. 讀完所有註解,先照類型分組:剪接(太緊/太鬆/刪掉)、字幕、特效/運鏡、音效。
2. **時間是 preview 的時間軸。剪接類的**(太緊/太鬆/某段刪掉)要先用 EDL 把 preview 時間**對回原始影片秒數**再改 range(絕不直接拿 preview 秒數當原始時間)。字幕/特效/運鏡類的就在那個 preview 時刻動手。
3. 所有改動一次做完,**只重跑一次 render**,再把新版本給他。
4. 還有要改的,他再審一輪 — 每輪都是「瀏覽器收齊 → 一次貼 → 一次改」。這樣通常 2-3 版就收斂,不會越改越發散。

有些註解會標「**(附畫記圖 審片-0-35.png,紅筆是位置)**」— 那是使用者在畫面上用畫筆圈了位置,圖下載在他的「下載」資料夾。這種(通常是字幕/特效位置類)先看那張圖:紅筆畫的地方就是他要的位置,比純文字準。請他把圖拉進來,或到下載資料夾讀 `審片-<時間>.png`。

### 7. 聲音(音效 + 背景音樂)— 一次 ffmpeg 混音,影片複製不重壓

**先讀 `素材庫/預設包/對照表.md`** — 那是「情緒/字眼 → 音效 + 運鏡」的準則表,五個基本音效(whoosh/pop/ding/riser/impact)跟三種調性的 BGM 都已經在 `素材庫/預設包/` 裡,含量好的建議音量。節奏提案時照表配,不用搜。母庫 `素材庫/` 還有 keyboard/brake/lofi 等使用者累積的檔。**預設包沒有合適的**才搜 heygen(要登入):`heygen audio sounds list --type sound_effects --query "..." --min-score 0.4 --limit 6`(音樂用 `--type music`)。
- **query 用英文比較好。** 中文 query 回來分數低、description 還會被 redact;英文 query 常 0.87+ 且描述看得懂。
- heygen 沒裝或沒登入時,不要在剪片剪到一半處理 — 先用預設包完成這支,片尾再問要不要裝(流程見文末「heygen 設定(選配)」)。
下載幾個候選讓**使用者用耳朵挑** — 語意分數不是耳朵。

**BGM/音效音量用「量測法」,不要背固定值。** 先 `ffmpeg -i f.mp3 -af volumedetect -f null -` 看音量,再依實測拉:目標 BGM 大概比人聲低 18–22dB(換算 volume 常落在 0.12–0.25,但**每首不一樣** — 這次一首 mean -25dB 要拉到 0.6、另一首 -15dB 要壓到 0.12)。使用者說「當背景就好」就往更低調。小聲的音效要放大(峰值 -18dB 的大概要 6 倍才聽得到)。

```bash
ffmpeg -y -i <專案>/工作檔/render_final.mp4 -i sfx1.mp3 -i bgm.mp3 -filter_complex "
[1:a]aformat=channel_layouts=stereo,volume=0.55,adelay=MS|MS[s1];
[2:a]atrim=0:DUR,afade=t=in:st=0:d=0.9,afade=t=out:st=END:d=1.4,volume=0.2[bgm];
[0:a][s1][bgm]amix=inputs=3:normalize=0[mix];[mix]alimiter=limit=0.97[aout]
" -map 0:v -map "[aout]" -c:v copy -c:a aac -b:a 192k <專案>/成品.mp4
```

**只有 BGM、沒有音效時,直接用 `bash $H/finalize.sh <專案>/工作檔/render_final.mp4 <bgm.mp3> <專案>/成品.mp4 [音量]`** — 它自動循環 BGM 蓋滿全片、進出 fade、限幅,影片流複製不重壓(~1 秒)。有音效要疊才用上面的完整 filter_complex。**完全沒有 BGM 也沒有音效時,成品就是 `cp 工作檔/render_final.mp4 成品.mp4`** — 三條路殊途同歸:成品.mp4 只從 render_final.mp4 產生、只有一份、在專案根目錄。**每次重 render 後都要重跑混音**(混音只存在輸出檔,不在 hyperframes 專案裡)。

背景音樂音量約 0.15-0.25(壓在人聲下面);短的曲子用 `-stream_loop` 循環。音效時間點用輸出時間軸的字詞秒數。用完問使用者要不要存進素材庫(見上面建議循環)。

**整片墊 BGM,不要有整段靜音**:BGM 從頭鋪到尾蓋滿整支,沒有死掉的空白(靜音很掉價)。人聲照樣壓在上面。(精準 LU 目標、逐段音量拉平屬進階,先不做 — 對新手 80 分幫助小又容易出錯。)

### 8. 完成檢查(硬性 — 成品出來後,照順序做完,一項都不能跳)

實測回報:少了這段,學員會「以為好了其實沒好」或「卡在中間,永遠沒走到回報」。成品.mp4 產生後:

1. **驗成品**:`ffprobe` 確認長度對(比對 render_final 的實際長度);抽 1-2 個影格確認字幕在畫面上;確認專案根目錄**只有一個**成品.mp4、沒有多的副本或「成品/」資料夾。
2. **整備審片區(最終版)**:把成品對應的 preview(或成品本身的複本)+ 最終 `字幕.json` 覆蓋進 `審片區/`,讓使用者之後隨時能拖進 審片.html 重看。這兩個檔從此**不歸 cleanup 刪**。
3. **一句話交付**:「完成了!成品在 我的影片/<專案名>/成品.mp4,長度 X 秒。想再檢查的話,審片區裡的兩個檔一樣拖進審片工具就能重看。」
4. **更新兩份筆記**(見下面收尾)。
5. **復盤 + 回傳**(見下面收尾)— 這步是流程的一部分,不是可有可無的加問。

另外,整個剪輯過程中養成**報站的習慣**:每完成一個階段,用一句話說「剛完成 X,接下來做 Y」— 使用者永遠知道現在在哪一步、還剩幾步。

## 收尾:更新兩份筆記(學習循環的終點)

**1. 跨專案:`KIT/我的剪輯偏好.md`** — 關於這位使用者、每支都適用的習慣:字幕風格(字型、大小、位置)、愛用的音樂調性、剪接節奏習慣、視覺鉤子與節奏工具偏好(愛用圖卡還實拍、highlight 要多還少、常說要或不要哪些工具)、常用的特效／品牌素材、任何他重複要求或明確不要的東西。簡短、具體、覆蓋掉「尚未記錄」。

**2. 專案內:`<專案>/工作檔/專案筆記.md`** — 只屬於這支影片、隔天回來要用的東西:這次確認過的**錯字對照表**、選定的**音樂/音效**、鉤子與卡片文案、節奏提案清單、目前到第幾版。這些原本只活在對話裡,session 一斷就沒了,隔天等於重問一輪。存成檔,下次一開場先讀它 → 零重問。(`tools/cleanup.sh` 會保留這份文字檔,只刪大媒體檔。)

這樣下次無論同一支或新的一支,一開場讀這兩份,就越來越懂他、越來越省。

**3. 主動復盤 + 回傳(幫工具包變好)** — 更新完筆記後,主動跟使用者做一輪簡短復盤,問 2-3 個具體的問題(不要開放式的「有什麼建議嗎」):
- 「這次哪一步卡最久 / 最煩?」
- 「有沒有哪個工具或流程你覺得應該更簡單?」
- 「有沒有想要但現在沒有的功能?」

把他的回答 + 你自己觀察到的問題(工具出錯、繞路、token 浪費)寫成 `KIT/錯誤回報/復盤-<YYYY-MM-DD>.md`,然後問一句:
> 要不要把這份復盤回傳給 Jake?他會拿來改進工具包,大家都受益。內容就是這份檔案,不會傳別的東西。

同意才傳:
```bash
bash $KIT/tools/freecut/helpers/send_report.sh --name "<學員名>" --project "<專案名>" --type "復盤" --content-file "KIT/錯誤回報/復盤-<日期>.md"
```
`--name` 用**使用者自己說的稱呼** — 先問一句「報告上要寫你是誰?名字或綽號都行」。**絕不拿 email 或其他個資當識別**(實測發生過:AI 自作主張填了使用者的 email)。送成功後把檔名加上 `已送-` 前綴(之後的 session 才不會重複問要不要送)。沒網路或傳失敗都沒關係(script 不會報錯卡住),檔案還在,照舊可以用 LINE 傳。**不同意就不傳,不要再問第二次。**

## heygen 設定(選配 — 使用者主動要更多音效/音樂才做)

預設包的音效跟 BGM 夠第一天用,所以第一次安裝不裝 heygen。使用者說想要更多選擇時才走這段,而且**先講清楚再動手**(這兩步是全流程最容易嚇到新手的):

1. **開場白**:「線上音效庫要裝一個小工具 + 用瀏覽器登入一次。過程有兩件事先說:裝的時候會跳一個權限確認 — 那是正常的,內容看一下按允許就好;登入會開網頁,而且**只有 5 分鐘**,所以我們先準備好再按。」
2. **安裝**(你跑):`curl -fsSL https://static.heygen.ai/cli/install.sh | bash` — 會被權限機制擋下請使用者按允許,這是設計不是故障。裝到 `~/.local/bin`,裝完 `source KIT/tools/env.sh` 才叫得到,驗證 `heygen --version`。
3. **登入**(倒數是從指令送出那秒開始算的,所以順序很重要):先問「準備好了嗎?按下去會出現一個網址,你要在 5 分鐘內打開它、登入 HeyGen、按授權。沒有 HeyGen 帳號也沒關係,那頁可以直接**用 Google 帳號登入/註冊**,最快」。他說好**才**跑 `HEYGEN_NO_BROWSER=1 heygen auth login --oauth`(這個環境變數讓它直接把 URL 印出來,不賭瀏覽器會不會自動開),URL 一出現**立刻**貼給他。注意:「他在 heygen 網站登入過」≠ CLI 有 token,一定要走這個流程。只用 OAuth,不用 API key 路線 — key 是機密(不能經過聊天視窗)而且計費走 API credits,跟訂閱是兩回事。
4. **驗證**:`heygen auth status`。逾時(`oauth: loopback timed out`)不用慌,再跑一次 — 這次他已經知道整個流程,通常來得及。連兩次逾時就先跳過(不影響剪片),記進回報。

## 互動原則(為什麼流程能又快又順)

1. 提案 → 確認 → 執行 → 給他看 → 修。絕不憑對聲音內容的猜測就重剪。
2. 使用者的耳朵 = 聲音的真相;波形 = 兩人共用的「指這裡」工具;原始影片時間 = 共用的座標系。
3. 使用者回報你驗證不了的東西(某個字重複了、音效沒對上),先用工具查(split_blobs、擷取影格)再改數字。
4. 輸出檔案編版本(`preview_v1..vN`),EDL 當唯一真相,舊 render 邊做邊刪。
5. **節奏工具是你的調色盤,不是丟給使用者的選單。** 由你判斷、主動提案,新手說「好」就有 80 分。每種工具有用量上限、擇機用、絕不每個都上、不疊在同一點。提案永遠針對這支的腳本量身出,不要固定模板、不要機械式 pattern — 目標是「一用就 80 分,但每個人的成品不長一樣」。
